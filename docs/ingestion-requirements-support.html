FR-ING-001

Structured File Ingestion

FormatStatusNotes
Comma-Separated Values (.csv) ✓ Supported Full support including column-header context scanning and NER-confirmed person-name detection.
Tab-Separated Values (.tsv) ✓ Supported Same column-context pipeline as CSV; delimiter auto-detected.
Microsoft Excel (.xlsx) ✓ Supported Cell values extracted from all worksheets via openpyxl (optional dep; skipped gracefully if not installed). Install with pip install pii-screener[docs].
Microsoft Excel (.xls) ✓ Supported Cell values extracted from all sheets via xlrd (optional dep). Install with pip install pii-screener[docs].
Apache Parquet (.parquet) ✓ Supported All column values from all row groups extracted via pyarrow (optional dep). Install with pip install pii-screener[docs].
Apache ORC (.orc) ✓ Supported All column values extracted via pyarrow (same dep as .parquet; optional). Install with pip install pii-screener[docs].
SQL dump files (.sql) ✓ Supported Scanned as plain text; all PII patterns applied. INSERT statement values are matched by regex.
Pipe-Separated Values (.psv) ✓ Supported Full support including column-header context scanning and NER-confirmed person-name detection; same pipeline as CSV/TSV. Delimiter auto-detected.
Fixed-width text files ✗ Not Supported No column-width schema support. Files would be scanned as raw text, losing column context and producing false positives.
FR-ING-002

Unstructured Document Ingestion

FormatStatusNotes
Microsoft Word (.docx) ✓ Supported Paragraphs and table cells extracted via python-docx (optional dep). Install with pip install pii-screener[docs].
Microsoft Word (.doc) ✓ Supported Text extracted from all OLE streams via olefile (optional dep); printable ASCII and UTF-16 LE runs captured. Install with pip install pii-screener[docs].
PDF — text-based ✓ Supported Text layer extracted via pdfminer.six (optional dep). Scanned PDFs (image-only) are not supported. Install with pip install pii-screener[docs].
PDF — scanned / image-based ✗ Not Supported Requires OCR pipeline: pdf2image + pytesseract or a cloud Vision API.
Plain text (.txt, .log, .md) ✓ Supported All three extensions are in the supported set and scanned with the full pattern + NER pipeline.
Rich Text Format (.rtf) ✓ Supported RTF control words stripped via striprtf (optional dep). Install with pip install pii-screener[docs].
HTML and XHTML ~ Partial .html/.htm scanned as raw text. PII in content is detected but HTML tags may generate noise. No DOM-aware parsing to skip <script>/<style> blocks.
XML ~ Partial .xml scanned as raw text. Attribute values and element content are matched but XML schema context is not used.
Email files (.eml) ✓ Supported Headers (From, To, Cc, Bcc, Subject…) and all text/* MIME parts extracted via Python stdlib email module — no extra dependencies required.
Email files (.msg) ✓ Supported Sender, recipients, subject, and body extracted via extract-msg (optional dep). Install with pip install pii-screener[docs].
FR-ING-003

Semi-Structured & Encoded Data

Format / EncodingStatusNotes
JSON ✓ Supported Scanned as text — all regex patterns work across nested, minified, and array-of-objects structures.
YAML ✓ Supported Both .yml and .yaml extensions supported. Config files and data feeds scanned fully.
AVRO ✓ Supported All record field values deserialised and scanned via fastavro (optional dep). Install with pip install pii-screener[docs].
Base64-encoded strings ✓ Supported Base64 blobs (≥24 chars) in scanned text are decoded and re-scanned for PII using all regex patterns (stdlib only, no extra deps).
URL-encoded strings ✓ Supported Consecutive percent-encoded sequences are decoded via urllib.parse.unquote and re-scanned inline (stdlib only).
JWT tokens — detection ✓ Supported JWT pattern (eyJ…eyJ…sig) detected and flagged as high-severity jwt_token.
JWT tokens — payload decode & scan ✓ Supported JWT payload (second segment) is Base64url-decoded and re-scanned for embedded PII claims (stdlib only).
Protobuf / MessagePack ✗ Not Supported Schema-dependent binary formats. Significant implementation complexity; not planned without schema provision.
FR-ING-004

Archive & Compressed File Traversal

FormatStatusNotes
ZIP ✓ Supported Every file entry is extracted via stdlib zipfile and scanned (text, binary docs, nested archives).
GZIP single-file (.gz) ✓ Supported Decompressed via stdlib gzip; inner file scanned by extension.
TAR / .tar.gz / .tgz ✓ Supported All members extracted via stdlib tarfile; plain .tar, gzip-compressed (.tar.gz/.tgz), and bzip2-compressed (.tar.bz2) all handled.
BZIP2 single-file (.bz2) ✓ Supported Decompressed via stdlib bz2; inner file scanned by extension.
7-Zip (.7z) ✗ Not Supported Requires third-party py7zr library; not included to avoid mandatory dependency.
Nested archives (configurable depth) ✓ Supported Archives within archives are recursed up to _MAX_ARCHIVE_DEPTH (default 10). Nested entries that are themselves archives trigger a recursive scan.
Depth-exceeded warning & skip ✓ Supported If nesting depth exceeds the limit, a warning is printed to stderr and that branch is skipped gracefully without crashing.
Password-protected ZIP ✗ Not Supported Encrypted entries raise an exception which is caught and silently skipped — no warning is printed. Content cannot be scanned without the password.
FR-ING-005

Image Processing

CapabilityStatusNotes
OCR scanning (JPEG, PNG, TIFF, BMP, WEBP, HEIC, GIF) ✗ Not Supported Image files are skipped entirely. Requires pytesseract + Pillow (or a cloud Vision API) for OCR.

FR-ING-006

Source Code Scanning

CapabilityStatusNotes
Python, JS/TS, Java, C#, C/C++, Go, Ruby, PHP, Bash/Shell, PowerShell ✓ Supported All extensions (.py .js .ts .jsx .tsx .java .cs .cpp .c .h .go .rb .php .sh .bash .zsh .ps1 .psm1) are in the supported set.
Infrastructure-as-Code (Terraform, Ansible, Dockerfile, K8s) ✓ Supported .tf, .tfvars, .hcl, and .yml/.yaml (Ansible/K8s) are all in the supported extension set. Dockerfile (no extension) is also scanned — the scanner’s extension filter allows any extensionless file to pass through (path.suffix == "" bypasses the allowlist check).
Config files (.env, .properties, appsettings.json, web.config) ~ Partial .env, .properties, and .json are all in the supported extension set (covering appsettings.json, appsettings.Development.json, etc.). web.config uses the .config extension which is not in the extension allowlist — these files are skipped.
String literal vs code logic distinction (low false positives) ~ Partial String-literal extraction and NER confirmation reduce false positives for person names. Regex-based patterns (email, SSN, etc.) match anywhere on the line including variable names.

FR-DET-001

UK-Specific PII Patterns

Data CategoryStatusNotes
Full name (NER) ~ Partial Presidio and spaCy NER detect person names in string literals and comments. Not fine-tuned on a UK English corpus; titles (Dr, Prof) not specifically handled.
Email address ✓ Supported Generic email regex matches all domains including .co.uk. Rule ID: pii_email.
UK phone number (+44 / 07xxx) ~ Partial Dedicated UK mobile rule added: pii_uk_phone_mobile covers 07xxx xxxxxx, +44 7xxx xxxxxx, 0044 7xxx xxxxxx. Generic E.164 (pii_phone_intl) also matches. UK landlines not explicitly covered.
UK postcode (SW1A 2AA) ✓ Supported Rule pii_uk_postcode covers all Royal Mail formats: AN, ANN, AAN, AANN, ANA, AANA. Medium severity, GDPR remediation guidance included.
National Insurance Number (NINO) ~ Partial Structural regex present (pii_ni_number). Check-character validation not implemented; invalid prefix combinations not fully excluded.
UK Passport Number ~ Partial Generic passport pattern (pii_passport) matches 7–9 alphanumeric chars. Not UK-specific; may produce false positives and miss UK format edge cases.
UK Driving Licence (MORGA753116SM9IJ) ✓ Supported Rule pii_uk_driving_licence: 5 surname chars (9-padded) + 6 DOB digits + 2 random alphanum + 1 digit + 2 letters = 16-char structural match. High severity.
NHS Number (3-3-4 with check digit) ✓ Supported Rule pii_nhs_number: 3-3-4 digit pattern with full modulus-11 check digit validation (_nhs_check). Invalid numbers are rejected. High severity, NHS DSP Toolkit references included.
Bank Sort Code & Account Number ✓ Supported Two rules: pii_uk_sort_code (keyword-contextual: requires sort[-_]code near the value) and pii_uk_account_number (keyword-contextual: requires account number/no/num). High severity, PSD2 and PCI DSS references.
Payment Card Number (PAN, Luhn-validated) ✓ Supported Visa, Mastercard, Amex, Discover, Diners patterns with Luhn validation. Rule ID: pii_credit_card.
BIC / IBAN (GB format) ~ Partial Generic IBAN regex present (pii_iban). Matches GB IBANs structurally but no IBAN check-digit (modulus 97) validation is performed.
Date of Birth ✓ Supported Two rules: pii_dob (YYYY-MM-DD / YYYY/MM/DD) and new pii_dob_uk (DD/MM/YYYY). Both UK and ISO date formats now covered.
IP Address (IPv4) ~ Partial IPv4 inside string literals detected (pii_ip_address). No context-dependent classification (public vs private ranges). Low severity.
IP Address (IPv6) ✓ Supported Rule pii_ipv6_address covers full 8-group colon-hex form. Compressed forms (::1, fe80::) not covered to avoid false positives.
Device MAC Address ✓ Supported Rule pii_mac_address matches colon-separated (aa:bb:cc:dd:ee:ff) and hyphen-separated (aa-bb-cc-dd-ee-ff) formats. Low severity, GDPR Recital 30 reference.
Biometric data ✗ Not Supported No biometric keyword/context detection implemented. Requires NER + domain-specific context markers.
Health / medical data (Special Category) ✗ Not Supported No ICD-10 code detection or medical context analysis implemented.
Racial / ethnic origin (Special Category) ✗ Not Supported No contextual analysis for protected characteristic data.
Political opinions (Special Category) ✗ Not Supported No contextual analysis for political opinion data.
Religious beliefs (Special Category) ✗ Not Supported No contextual analysis for religious belief data.
Sexual orientation / life (Special Category) ✗ Not Supported No contextual analysis for sexual orientation data.
FR-DET-002

Detection Techniques

TechniqueStatusNotes
Regular Expression Engine ✓ Supported Curated regex pattern library covering 15+ PII categories. Luhn validation implemented for payment cards. UK-specific tuning is partial (see FR-DET-001).
Named Entity Recognition (NER) ~ Partial Presidio and spaCy supported as optional backends. General English models used — not fine-tuned on a UK English corpus. Organisations, locations, and dates not explicitly targeted beyond person names.
Context-Aware Classification ~ Partial NER confidence scoring and string-literal extraction reduce false positives. No token-window analysis around candidate matches (e.g. no NHS number context check).
Dictionary Lookups (names, places) ✗ Not Supported No forename, surname, or UK place-name dictionary. NER partially compensates but dictionary lookups are not implemented.
Entropy Analysis ✗ Not Supported High-entropy string detection (e.g. API keys, encoded tokens) is not implemented. Gitleaks (external scanner) covers some of this for secrets.
Checksum Validation (Luhn, NINO, NHS) ~ Partial Luhn mod-10 algorithm implemented for payment cards (pii_credit_card). NHS modulus-11 check digit fully implemented (_nhs_check) — invalid NHS numbers are rejected before a finding is raised. NINO check-character validation is not implemented; invalid prefix combinations are not fully excluded.
FR-DET-003

Configurable Detection Profiles

Profile / FeatureStatusNotes
Standard scan profile ~ Partial All detectors run by default. There is no formally defined "standard" profile that limits to common PII only for faster performance.
Deep scan profile (all categories, full ML) ✗ Not Supported No separate deep-scan mode. Special category data (health, biometrics, etc.) is not detected regardless of profile.
Custom profile (user-defined detector subset) ~ Partial Rule IDs can be suppressed globally or per-scanner via suppress.txt or sensitive-scanner.yaml. Specific scanners can be disabled with --scanners. No named profile system exists.
Allowlisting (known-clean fields / false-positive suppression) ~ Partial Rule-ID-level suppression and inline # noscan comments supported. Column-name exclusion via _SKIP_NAME_COL_RE regex for CSV. No declarative allowlist for specific field values or field paths.
FR-CLS-001

Finding Classification

Metadata FieldStatusNotes
PII category (taxonomy) ✓ Supported Finding.category populated for every finding (e.g. pii_email, pii_ssn, hardcoded_password).
Sensitivity tier (Standard / High / Special Category) ~ Partial Finding.severity uses critical/high/medium/low/info scale — not the Standard/High/Special Category UK GDPR taxonomy required.
Regulatory article(s) implicated (UK GDPR Art. 9 etc.) ~ Partial Finding.regulations is populated by every scanner via RegulationEngine.lookup(category) (e.g. ["UK-GDPR-PD", "PSR-2017"]). Rendered as regulation-ID badges in the console, Markdown, and HTML findings tables. GDPR Article numbers ARE modelled in config/regulations.yaml (UK-GDPR-PD = Article 4(1), UK-GDPR-SC = Article 9(1), UK-GDPR-51F = Article 5(1)(f)) and displayed as reg-article-num in the HTML “Applicable Regulations” summary cards. Article numbers are not shown inline per-finding in the findings table — only the regulation ID badge is rendered there.
Detection confidence (Certain / Probable / Possible) ~ Partial Finding.confidence (float 0.0–1.0) is set per scanner, boosted by the orchestrator when multiple scanners agree (+8% per agreeing scanner, capped at 99%). The HTML report renders colour-coded confidence pill badges via the --show-confidence flag. JSON output includes the numeric confidence value on every finding (serialised by model_dump_json). Not exposed as the required Certain / Probable / Possible tier; confidence is not surfaced in the Markdown report format.
Source file path and line reference ✓ Supported Finding.file (relative path) and Finding.line (1-based) present on every finding.
Column reference ~ Partial CSV scanner records the column header name in the message but no structured column field on Finding.
Character offset ✗ Not Supported Character offset within the line is not stored on Finding.
Contextual snippet (configurable, redacted) ~ Partial Finding.match stores the matched value (redacted by default). No surrounding context window (±50 chars) is stored or rendered.

FR-RPT-005

Audit Trail

ElementStatusNotes
Immutable audit trail of scan activities ✗ Not Supported No audit log. Scan results are written to report files only; no append-only event log is maintained.
Operator identity logging ✗ Not Supported No authentication or user identity capture.
Timestamp of scan ✓ Supported Report.scanned_at records the scan timestamp and is included in all report formats.
Scan configuration logged ~ Partial Report records project_name, target_path, scanners_run, and tier_used. Full ScanConfig (excludes, suppressions, flags) is not persisted in the report output.
Findings count logged ✓ Supported ScanSummary.total and per-severity counts recorded in every report.
Remediation actions logged ~ Partial The obfuscate command performs in-place source-file replacements (with file backups and a rollback command to undo). NFR-P-001–006

Performance Non-Functional Requirements

ID Requirement Target Status Notes
NFR-P-001 Structured file throughput ≥ 1 GB/min (4-core, 16 GB RAM) ~ Partial With the regex-only layer active, throughput on plain-text and code files is likely to approach this target — file I/O is async (aiofiles) and patterns are pre-compiled. When the NER layer is active (Presidio + spaCy en_core_web_sm), every chunk passes through an in-process neural model, reducing throughput materially below 1 GB/min. No formal benchmark has been run against a 4-core 16 GB node.
NFR-P-002 Unstructured document throughput ≥ 500 pages/min (text-based PDF) ✗ Not Met PDF extraction uses pdfminer.six, which is synchronous and single-threaded. Typical throughput is 1–8 pages/sec (60–480 pages/min) depending on page complexity — unreliably below the 500 pages/min target. Scanned (image-only) PDFs are not supported at all (see FR-ING-005 and NFR-P-003). Would require a faster backend (e.g. pymupdf) and extraction in a ProcessPoolExecutor.
NFR-P-003 Image OCR throughput ≥ 60 images/min at 300 DPI ✗ Not Supported No OCR capability exists. pytesseract, easyocr, Pillow, and Tesseract are absent from the dependency tree. Image formats (.png, .jpg, .tiff, etc.) are silently skipped during traversal. NFR cannot be assessed against a throughput target as the capability does not exist — see FR-ING-005.
NFR-P-004 Database scan rate ≥ 100,000 rows/min per connection ✗ Not Supported No live database connectivity (see FR-ING-007). The db_conn_string rule detects credentials embedded in source code — it is not a database content scanner. Flat-file exports (CSV, Parquet, ORC, Avro) are supported but no row-throughput benchmark exists and the 100,000 rows/min target has not been validated. Requires a dedicated connector with connection pooling and batched reads.
NFR-P-005 Report generation time Executive report within 60 seconds of scan completion ✓ Met Report generation is entirely in-memory. The Jinja2 Environment is cached at module level (no recompilation) and the RegulationEngine is lazy-initialised and cached. All four output formats (HTML, Markdown, JSON, console) complete in well under 5 seconds in practice, even for scans with thousands of findings.
NFR-P-006 Horizontal scalability Linear scaling to 64 parallel worker nodes ✗ Not Met The tool is a single-node, single-process application. Concurrency is provided by asyncio and by running scanner subprocesses in parallel — good utilisation of one multi-core machine, but not distributed across hosts. No work queue, coordinator/worker topology, or shared state store exists. Would require a task-queue layer (e.g. Celery + Redis, or Ray) partitioning the file tree across workers, with a central coordinator merging and deduplicating results.
