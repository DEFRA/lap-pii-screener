## Basic usage

```powershell
sensitive-scanner scan <path>
```

`<path>` must be a directory. The scanner walks it recursively. Results print to the terminal with colour coding.

Examples:

```powershell
# Scan a project folder
sensitive-scanner scan C:\Github\MyProject

# Save as an HTML report
sensitive-scanner scan C:\Github\MyProject --format html --output report.html

# Save as Markdown (for Confluence, GitHub, VS Code preview)
sensitive-scanner scan C:\Github\MyProject --format markdown --output report.md

# Save as JSON (for downstream processing)
sensitive-scanner scan C:\Github\MyProject --format json --output report.json
```

## Choosing scanners

By default all available scanners run. To run a specific subset:

```powershell
sensitive-scanner scan C:\Github\MyProject --scanners gitleaks,semgrep
```

Available scanner names:

| Name | What it does |
|---|---|
| `gitleaks` | Secret pattern matching (AWS, GitHub, Stripe keys, etc.) |
| `semgrep` | Code-structure-aware analysis (OWASP Top 10, hardcoded secrets) |
| `presidio` | Custom PII detection (emails, phone numbers, NHS, NI, DoB, names) |
| `sonarqube` | Deep inter-procedural analysis (requires SonarQube running) |

Scanners that are not installed are skipped automatically without error. The active tier is shown when you run `sensitive-scanner status`.

Multiple scanners can independently find the same issue. When they do, the finding appears once in the report with all scanner names listed and a boosted confidence score.

## Output and report formats

```powershell
sensitive-scanner scan <path> --format <format> --output <file>
```

| Format | Flag | Best for |
|---|---|---|
| Console (default) | `--format console` | Immediate review in the terminal |
| HTML | `--format html` | Sharing, email attachments, management reports |
| Markdown | `--format markdown` | Confluence, GitHub, VS Code preview |
| JSON | `--format json` | Feeding into other tools, CI artefact storage |

If `--output` is omitted, the report prints to stdout. For HTML and Markdown this writes raw content, so use `--output` to save to a file.

For detailed information about each format see [Reports](reports.md).

### Showing full matched values

By default, matched values are redacted in all report formats (`john****`, `AKIA****`). To see the full value:

```powershell
sensitive-scanner scan <path> --show-secrets
```

> **Use with care.** Reports generated with `--show-secrets` contain the actual sensitive values. Do not share them or commit them to source control.

## Controlling what is scanned

### Default exclusions

The following are excluded automatically from every scan:

- `.git`, `.vs`, `.idea`, `.vscode` — version control and editor directories
- `node_modules`, `dist`, `build`, `out`, `target` — dependency and build output directories
- `__pycache__`, `.venv`, `env`, `venv` — Python virtual environments and caches
- `coverage`, `htmlcov` — test coverage output
- `bin`, `obj` — compiled binary output
- Binary file types (`.exe`, `.dll`, `.zip`, `.png`, `.jpg`, etc.)
- Files larger than 2 MB
- Any report file previously generated by PII Screener in the scan root

### Excluding directories, patterns, and files

**Via CLI flag:**

```powershell
# Exclude additional directories by name
sensitive-scanner scan <path> --exclude "coverage,test-fixtures,generated"

# Exclude using glob patterns
sensitive-scanner scan <path> --exclude "**/*.min.js,src/generated/**"
```

Entries without wildcards are matched against directory names anywhere in the path tree. Entries containing `*`, `?`, or `/` are treated as glob patterns matched against the file's relative path.

**Via config file:**

```yaml
exclude:
  directories:
    - coverage
    - migrations
    - docs

  patterns:
    - "**/*.min.js"
    - "**/*.bundle.js"
    - "src/generated/**"

  files:
    - "tests/fixtures/sample_data.json"
    - "docs/examples/config.py"
```

### .scannerignore file

Place a `.scannerignore` file in the root of the directory you are scanning. It uses the same syntax as the `--exclude` flag: one entry per line, comments with `#`.

```
# Ignore test data
tests/fixtures/**
tests/data/**

# Ignore generated code
src/generated/
**/*.g.cs

# Ignore specific files
docs/examples/config.py
```

The `.scannerignore` file is project-specific and should be committed alongside your code.

### Scanning Git history

By default only the current working tree is scanned. To include deleted files and previous commit history (useful for checking whether secrets were ever committed, even if later removed):

```powershell
sensitive-scanner scan <path> --history
```

This passes Gitleaks into Git log mode and scans every blob in the repository's history. It can be significantly slower on large repositories with long histories.

## Suppressing false positives

A false positive is a finding that the scanner flags as sensitive but is not actually a real issue — for example, a placeholder email address `user@example.com` in a test fixture.

### One-time suppression

Pass a comma-separated list of rule IDs to suppress for this scan only:

```powershell
sensitive-scanner scan <path> --suppress "pii_email,pii_phone"
```

The rule ID is shown in the **Rule** column of every finding. You can also reference OWASP or CWE identifiers listed in the References column.

### Persistent global suppression

To suppress a rule on every future scan, add it to `config\suppress.txt` (the global file at the PII Screener installation):

```
# Test fixtures — not real PII
pii_email

# Generated config — not a real connection string
db_conn_string
```

Blank lines and lines starting with `#` are ignored.

You can also create a `suppress.txt` in the root of your project folder. Both files are read and merged: the global file applies everywhere, the project file applies only to that project.

### Per-scanner suppression

To suppress a rule only for a specific scanner (useful when one scanner generates noise on a particular rule that another scanner covers correctly), use section headers in `suppress.txt`:

```
# Global — all scanners
CWE-798

[presidio]
pii_email
pii_phone

[gitleaks]
aws-access-token

[semgrep]
python.lang.security.audit.formatted-sql-query

[sonarqube]
secrets:S6706
```

Lines before any section header are global. Lines under a `[scanner]` header apply only to that scanner. The section header must use the exact scanner name: `gitleaks`, `semgrep`, `presidio`, `sonarqube`.

You can also specify per-scanner suppressions in `sensitive-scanner.yaml`:

```yaml
suppress_by_scanner:
  presidio:
    - pii_email
    - pii_phone
  gitleaks:
    - aws-access-token
  semgrep:
    - python.lang.security.audit.formatted-sql-query
  sonarqube:
    - secrets:S6706
```

### Inline code suppression

To suppress a specific line in your source code, add a `noscan` comment:

```python
# Suppress all rules on this line
test_email = "user@example.com"  # noscan

# Suppress only a specific rule on this line
test_email = "user@example.com"  # noscan: pii_email
```

This works with any comment style: `//`, `#`, `--`, `<!-- -->`.

## Scan configuration file

For projects you scan regularly, create a `sensitive-scanner.yaml` in the project root. The scanner picks it up automatically. CLI flags always override values from the config file.

```yaml
# sensitive-scanner.yaml

# Scanners to run (defaults to all available)
scanners:
  - gitleaks
  - semgrep
  - presidio
  - sonarqube

# Output format: console, html, markdown, json
format: html

# Report file (relative to cwd when you run the scan)
output: scan-report.html

# Project name in the report header and SonarQube project key
project_name: MyProject

# Exit code 2 if any finding at or above this severity is found
# Useful for CI. Values: critical, high, medium, low
fail_on: high

# Include full git commit history
include_git_history: false

# Show full matched values instead of redacting
show_secrets: false

# Skip code comments
skip_comments: false

# Suppress specific rule IDs (global — all scanners)
suppress:
  - pii_email
  - db_conn_string

# Per-scanner suppressions
suppress_by_scanner:
  presidio:
    - pii_phone
  sonarqube:
    - secrets:S6706

# Exclusions
exclude:
  directories:
    - coverage
    - migrations
  patterns:
    - "**/*.min.js"
    - "src/generated/**"
  files:
    - "tests/fixtures/sample_data.json"
```

Point to a config file in a non-standard location with `--config`:

```powershell
sensitive-scanner scan <path> --config C:\configs\my-scanner.yaml
```

## Confidence scoring

Each finding has a confidence score (0–100%) representing how certain the scanner is that the finding is a genuine issue. It is not shown by default.

To include confidence in an HTML report:

```powershell
sensitive-scanner scan <path> --format html --output report.html --show-confidence
```

| Colour | Range | Meaning |
|---|---|---|
| Green | ≥ 85% | Strong evidence — highly likely a real finding |
| Amber | 65–84% | Moderate evidence — worth reviewing |
| Red | < 65% | Weak signal — context review needed |

When multiple scanners independently find the same issue, the confidence is boosted by 8% per additional scanner (capped at 99%). This means a finding caught by both Gitleaks and Semgrep carries higher confidence than one found by only one scanner.

The report also shows a summary card counting how many findings fall below 65% when `--show-confidence` is active.

## Per-file reports

Write one report file per scanned source file instead of a single combined report:

```powershell
sensitive-scanner scan <path> --format json --per-file --output-dir ./scan-reports
```

The output directory mirrors the scanned directory structure. For example, `src/auth/config.py` would produce `scan-reports/src/auth/config.py.json`.

If `--output-dir` is not supplied with `--per-file`, reports are written to `./scan-reports` by default.

## CI pipeline integration

Use `--fail-on` to make the scan exit with code 2 when findings meet or exceed a severity threshold:

```powershell
# Fail the build if any critical or high findings are found
sensitive-scanner scan . --fail-on high --format json --output scan-results.json
```

Exit codes:

| Code | Meaning |
|---|---|
| 0 | Scan completed, no findings at or above the threshold |
| 1 | Error (bad arguments, path not found, etc.) |
| 2 | Scan completed, findings found at or above `--fail-on` threshold |

GitHub Actions example:

```yaml
- name: Run PII/secrets scan
  run: |
    sensitive-scanner scan . --fail-on high --format json --output scan-results.json
  continue-on-error: false

- name: Upload scan results
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: pii-scan-results
    path: scan-results.json
```

## Understanding results

### Severity levels

| Severity | Meaning | What to do |
|---|---|---|
| **Critical** | Exposed secret or PII almost certainly real | Fix immediately |
| **High** | Strong likelihood of a real issue | Fix before sharing the code |
| **Medium** | Could be an issue — review the context | Investigate and fix or suppress |
| **Low** | Weak signal — probably fine but worth a look | Review at your convenience |

### Finding fields

| Field | What it means |
|---|---|
| **ID** | Unique identifier for this finding — used in obfuscation and edit commands |
| **Severity** | How serious the issue is |
| **Category** | Type of issue: `pii_email`, `aws_access_key`, `hardcoded_password`, etc. |
| **File** | File path relative to the scan root |
| **Line** | Line number in that file |
| **Match** | Redacted preview of the matched value (e.g. `john***`) |
| **Rule** | Rule ID that triggered this finding — use this in `--suppress` |
| **Scanners** | Which scanner(s) detected this |
| **Fix** | Step-by-step remediation instructions |
| **Regulations** | Applicable regulations: GDPR, PCI DSS, PSR 2017, etc. |

## All scan options reference

```
sensitive-scanner scan <path> [OPTIONS]

Arguments:
  path                     Directory to scan (required)

Options:
  -s, --scanners TEXT      Comma-separated scanners: gitleaks,semgrep,presidio,sonarqube
  -f, --format TEXT        Output format: console (default), html, markdown, json
  -o, --output PATH        Write report to this file (default: stdout)
  -p, --project TEXT       Project name for report header and SonarQube project key
      --history            Include git commit history in the scan
      --fail-on TEXT       Exit code 2 if findings at or above severity: critical|high|medium|low
  -e, --exclude TEXT       Additional dirs/patterns to exclude (comma-separated)
      --show-secrets       Show full matched values instead of redacting
      --show-confidence    Add confidence column to HTML report
      --skip-comments      Do not scan code comment lines
      --suppress TEXT      Comma-separated rule IDs to suppress from results
  -c, --config PATH        Path to YAML config file (defaults to sensitive-scanner.yaml in scan dir)
      --per-file           Write one report file per source file
      --output-dir PATH    Directory for per-file reports (implies --per-file)
      --session PATH       Obfuscation session file to include in HTML report
```
